# **TrainBot PRD 최종본**

/김천구미 ↔ 동탄 주간 예매 어시스턴트 (NAS Docker + React UI + Telegram)

---

## **0) 문서 메타**

- 버전: v1.0 (최종본)

- 작성 기준: 2026-03-02 (Asia/Seoul)

- 대상: 1인 운영(개인) + 최대 4인 공동 사용 가능

## **1) 제품 개요**

### **1.1 문제 정의**

김천대학교(김천) 통학으로 인해 매주 반복되는 **김천구미↔동탄 왕복 열차 예매**가 필요하다. 수동 검색/예매는 번거롭고, 매진·취소표 등 변수가 많아 반복 부담이 크다.

### **1.2 목표**

- 매주 반복되는 왕복 예매 업무를 **추천/알림 중심으로 자동화**

- **SRT 직행을 최우선**으로 추천하고, 필요 시 **SRT+KTX 환승** 대안도 제공

- 선호 시간대를 “요일 전체 기준 + N시 이후(8/9/12…)”로 간단히 설정 가능

- 결과를 **텔레그램으로 자동 발송**

- NAS에서 Docker로 안정적으로 운영

- (옵션) **결제까지 수행하는 자동예매 기능을 구현**하되, **기본 비활성**이고 사용자가 명시적으로 활성화해야만 동작

### **1.3 핵심 가치**

- 매주 “검색→판단→알림”을 자동화하여 예매 준비 시간을 최소화

- 직행 우선 + 환승 대안 제공으로 선택 폭 확대

- 운영/관리(UI) 중심으로 NAS에 올려두고 지속 사용 가능

## **2) 범위 정의**

### **2.1 포함(In Scope)**

- React Web UI (설정/실행/스케줄/로그/사용자관리)

- 추천 엔진(직행 우선 + 환승 후보 생성 + 점수화/정렬)

- 선호 시간대 규칙: 요일별 **earliest_after(‘N시 이후’)** 필터

- 텔레그램 알림 및 중복 방지(dedupe)

- 수동 실행 및 스케줄 실행(등록/활성/비활성/삭제)

- 인증/권한:

  - **카카오 로그인(OAuth)만 허용**

  - **최초 가입자 = 관리자(Admin)**

  - **최대 4인까지만 가입 허용**

  - **관리자 승인제(가입 신청 → 승인 후 사용 가능)**

- 자동예매(결제 포함) 옵션 기능: **구현 대상**, 기본 비활성, 강력한 가드레일 포함

### **2.2 제외(Out of Scope)**

- 자동예매를 기본값으로 활성화

- 캘린더 자동 등록(2차)

- 개인별 모델 학습/지능형 추천 고도화(2차)

- 모바일 앱(웹 UI로 대체)

## **3) 사용자/권한/가입 정책**

### **3.1 사용자 역할**

- **Admin(관리자)**

  - 사용자 승인/거절/비활성화

  - 시스템 설정/스케줄/로그 전체 접근

  - (기본 정책) 자동예매(auto) 활성 권한 보유

- **Member(일반 사용자)**

  - 추천 결과 열람, 수동 실행(정책에 따라 제한 가능), 텔레그램 재전송

  - 시스템 핵심 설정/자동예매 활성화 권한 없음(기본)

### **3.2 인증 방식**

- **카카오 로그인만** 제공

- 이메일/비밀번호/기타 OAuth 제공하지 않음

### **3.3 가입/승인 플로우**

1. 사용자 카카오 로그인 → 가입 신청 생성 PENDING

2. 최초 가입 성공 사용자는 자동으로 ADMIN + ACTIVE

3. 그 외 사용자는 PENDING 상태로 생성

4. 관리자가 UI에서 승인하면 ACTIVE로 전환되어 사용 가능

5. 거절 시 REJECTED, 비활성화 시 DISABLED

### **3.4 사용자 정원 제한**

- ACTIVE 사용자 기준으로 **최대 4명**

- 4명 도달 시 신규 가입 신청은:

  - 기본: “가입 불가(정원 초과)” 처리

  - (선택) PENDING은 만들되 승인 불가 상태로 표시(정책 선택 가능)

## **4) 기능 요구사항 (Functional Requirements)**

## **4.1 노선/역 설정 (확장 가능)**

- 기본 노선(Primary)은 고정으로 시작:

  - 김천구미 ↔ 동탄

- 확장 가능한 설계:

  - alternatives.enabled가 ON이면 대체역을 후보 생성에 포함

  - 대체역은 “환승 후보” 또는 “대체 도착/출발 제안”에 사용 가능

- 역 리스트는 내부 코드(id) + 표시명(name)을 분리하여 확장 가능해야 함

## **4.2 선호시간대 규칙 (“요일 전체” + “N시 이후”)**

- 방향(up/down) + 요일별로 earliest_after 설정 가능

  - 예: 8/9/12/18시 이후 등

- 규칙 적용:

  - 해당 요일 00:00\~23:59 전체에서 earliest_after 이후의 열차만 후보로 허용

- 기본값(초기 설정):

  - 상행(김천→동탄): 금 18시 이후, 토 8시 이후

  - 하행(동탄→김천): 일 12시 이후

## **4.3 추천 후보 생성**

### **4.3.1 직행 후보**

- **SRT 직행**을 최우선으로 후보 생성 및 출력

- 직행 후보는 출력 Top N(기본 5)

### **4.3.2 환승 후보**

- 환승 후보는 allow_transfer=true일 때만 생성

- 기본 제한:

  - 최대 환승 횟수: 1

  - 최소 환승 여유: 20분

- 환승 후보 출력 Top N(기본 3)

### **4.3.3 후보 수 제한**

- 방향별 후보 생성 최대치(예: 30개)

- 과도한 조합 폭발 방지(대체역/환승 모두 켠 경우 필수)

## **4.4 스코어링/정렬**

- 기본 우선순위:

  1. SRT 직행(가산점)

  2. 시간 조건(earliest_after) 충족

  3. 총 소요시간 짧음

  4. 환승 횟수 적음

  5. 환승 대기시간 짧음

- 가중치(설정 가능 항목):

  - direct_bonus

  - total_time_penalty_per_min

  - transfer_penalty

  - transfer_wait_penalty_per_min

## **4.5 결과 출력(UI)**

- 방향 탭: 상행/하행

- 섹션:

  - SRT 직행 Top N (항상)

  - 환승 Top N (옵션)

- 항목 카드 표시:

  - 출발역/도착역

  - 출발/도착 시각

  - 총 소요시간

  - (환승 시) 환승역/횟수/환승 대기시간

  - 태그(직행/최단/환승1/대체역/조건부합 등)

- 액션:

  - 검색값 복사(날짜/역/열차/시간)

  - 텔레그램 재전송

  - (가능 시) 링크 제공

## **4.6 텔레그램 알림**

- 발송 내용:

  - 제목: 방향 + 날짜 + 노선

  - 본문: 직행 후보 → 환승 후보 순서

  - 하단: 적용 조건 요약(컷오프/환승/대체역 등)

- 중복 방지(dedupe):

  - 동일 결과 해시 기반으로 일정 시간(예: 180분) 내 중복 발송 금지

- 테스트 메시지 전송 기능 제공

## **4.7 실행/스케줄**

### **4.7.1 수동 실행**

- UI에서 “이번 주/다음 주” 선택 후 **\[지금 검색\]** 클릭 시 실행

### **4.7.2 스케줄 실행**

- 스케줄 기능:

  - 등록/활성/비활성/삭제

  - 크론 프리셋(예: 매주 화 09:00)

  - 스케줄별 this_week | next_week 선택 가능

- 실행 중복 방지:

  - 동일 시간대에 실행 요청이 중복되면 하나만 수행(락)

## **4.8 자동예매(결제 포함) 옵션 기능 — “구현하되 옵션 활성화”**

> **요구사항에 명시적으로 포함**:

> “결제를 대신 수행하는 형태의 자동예매도 구현은 하되, 옵션으로 활성화하도록 한다.”

### **4.8.1 모드 정의**

- assist(기본): 추천/알림/복사/링크

- auto(옵션): 검색→좌석확보→결제까지 자동 수행(플러그인/모듈 분리)

### **4.8.2 활성화 정책(필수)**

- 기본값 auto.enabled=false

- UI에서 아래 절차를 모두 거친 경우에만 활성화:

  1. 리스크 안내문 표시(고위험 기능)

  2. 체크박스 2개 이상(이해/책임)

  3. 2단계 확인(지정 문구 직접 입력)

- 권한 정책(기본):

  - **관리자만 auto 활성/설정 변경 가능**

### **4.8.3 제한/안전장치(필수)**

- 예산/횟수 제한:

  - 1회 최대 결제 금액

  - 주간 최대 결제 횟수

  - 실행당 최대 시도 횟수

- 위험 신호 감지 시 즉시 중단 + 텔레그램 알림:

  - 로그인 실패 반복, 차단/대기열 신호, 응답 포맷 변경, 결제 오류 등

- 민감정보 처리:

  - 결제 관련 민감정보는 설정/로그 저장 금지(환경변수/안전 저장소 사용)

### **4.8.4 구현 분리 원칙**

- 기본 배포본에는 assist 완성

- auto는 플러그인 인터페이스로 연결되며, 플러그인 미설치 시 auto 기능은 UI에서 비활성(또는 “미설치” 표시)

## **9) 수용 기준 (Acceptance Criteria)**

1. 카카오 로그인만 가능하며 최초 가입자는 자동으로 관리자/활성 상태가 된다.

2. 이후 가입자는 관리자 승인 전까지 PENDING이며 시스템 기능 접근이 제한된다.

3. ACTIVE 사용자 수가 최대 4명으로 제한되고 UI에 표시된다.

4. 요일별 “N시 이후” 설정이 가능하며 추천 결과에 정확히 반영된다.

5. 추천 결과는 SRT 직행이 우선이며, 환승 허용 시 환승 후보가 별도 섹션으로 제공된다.

6. UI에서 수동 실행/스케줄 실행이 가능하고 실행 결과가 저장/조회된다.

7. 텔레그램으로 추천이 발송되며 동일 결과 중복 발송이 지정 시간 동안 제한된다.

8. 자동예매(결제 포함) 기능은 **구현 대상**이며, **기본 비활성**이고 사용자(관리자)의 명시적 활성화 절차를 거쳐야만 동작한다(가드레일 포함).

9. 로그/감사 기능을 통해 실행·설정·승인 관련 이벤트가 추적 가능하다.

## **10) 용어 정의**

- **상행(up)**: 김천구미 → 동탄

- **하행(down)**: 동탄 → 김천구미

- **컷오프(earliest_after)**: 해당 요일 전체에서 “N시 이후만 허용”하는 시작 시각

- **assist 모드**: 추천/알림/복사/링크 중심(기본)

- **auto 모드**: 결제까지 자동 수행(옵션, 기본 비활성, 플러그인)
