## 1. 문서 개요

### 1.1 목적

이 문서는 **TrainBot** 서비스의 비즈니스 동작 규칙을 한곳에 모아 정의한다.

**이 문서의 대상 독자:**

| 독자 | 활용 방법 |
| --- | --- |
| AI 코딩 도구 (Claude Code 등) | 구현 시 비즈니스 규칙 참조. 예시의 "상황 → 결과"를 테스트 케이스로 활용 |
| 개발 (본인) | 비즈니스 로직 구현의 기준 문서 |
| QA | 테스트 케이스 도출의 기반 |

### 1.2 관련 문서

| 문서명 | 관계 |
| --- | --- |
| 서비스 기획서 (SPD) | 본 문서의 상위 문서. 서비스 컨셉, MVP 스코프 정의 |
| 요구사항 명세서 (SRS) | 본 문서의 정책이 SRS의 기능 요구사항(FR)에서 참조됨 |

## 3. 인증/권한 정책

### 3.1 목적

소규모(최대 4인) 서비스에 적합한 인증 체계를 제공하고, 역할에 따라 시스템 기능 접근을 제어한다.

### 3.2 규칙

#### BP-AUTH-001: 인증 방식

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| AUTH-001-1 | **카카오 로그인(OAuth)만** 제공한다. 이메일/비밀번호/기타 OAuth는 제공하지 않는다 |  |
| AUTH-001-2 | 카카오 로그인 성공 시 서버에서 세션 또는 JWT를 발급한다 |  |
| AUTH-001-3 | 로그아웃 시 세션/토큰을 무효화한다 |  |

#### BP-AUTH-002: 역할 및 권한 (RBAC)

| 역할 | 설명 | 주요 권한 | 비고 |
| --- | --- | --- | --- |
| Admin (관리자) | 최초 가입자, 시스템 전체 관리 | 설정 변경, 스케줄 관리, 사용자 승인/거절/비활성화, auto 모드 활성/설정, 수동 실행, 로그 전체 접근 | 최소 1명 |
| Member (일반 사용자) | 관리자 승인 후 활성화 | 추천 결과 열람, 수동 실행(정책에 따라 제한 가능), 텔레그램 재전송 | 시스템 핵심 설정/auto 활성화 권한 없음 |

#### BP-AUTH-003: 권한 매트릭스

| 기능 | Admin | Member | 미인증/PENDING |
| --- | --- | --- | --- |
| Dashboard 조회 | O | O | X |
| 캘린더 관리 (상태 변경/메모) | O | O | X |
| 수동 실행 (지금 검색) | O | O (정책에 따라 제한 가능) | X |
| 결과 조회 | O | O | X |
| 설정 변경 | O | X | X |
| 스케줄 관리 | O | X | X |
| 사용자 승인/거절/비활성화 | O | X | X |
| auto 모드 활성/설정 | O | X | X |
| 로그 전체 조회 | O | X | X |
| 텔레그램 재전송 | O | O | X |

### 3.3 예시

**정상 케이스:**

```
상황: 사용자가 카카오 로그인을 완료한다 (기존 ACTIVE 계정)
결과: 세션/토큰이 발급되고 Dashboard로 이동한다
```

**경계 케이스:**

```
상황: PENDING 상태의 사용자가 로그인한다
결과: 로그인은 성공하지만 "관리자 승인 대기 중입니다" 안내 화면을 표시한다. 시스템 기능에 접근할 수 없다
```

**에러 케이스:**

```
상황: REJECTED 또는 DISABLED 상태의 사용자가 로그인을 시도한다
결과: "계정이 비활성화되었습니다. 관리자에게 문의하세요." 메시지를 표시한다
```

## 4. 사용자 가입/승인 정책

### 4.1 목적

소규모 운영 환경에서 사용자를 관리자 승인제로 통제하고, 정원 초과를 방지한다.

### 4.2 규칙

#### BP-USER-001: 가입/승인 플로우

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| USER-001-1 | 사용자가 카카오 로그인 시 기존 계정이 없으면 가입 신청(PENDING)이 자동 생성된다 |  |
| USER-001-2 | **최초 가입 사용자**는 자동으로 **ADMIN + ACTIVE** 상태가 된다 | 시스템에 ACTIVE 사용자가 0명일 때 |
| USER-001-3 | 이후 가입자는 **PENDING** 상태로 생성되며, 관리자 승인 전까지 시스템 기능에 접근할 수 없다 |  |
| USER-001-4 | 관리자가 승인하면 **ACTIVE**로 전환된다 |  |
| USER-001-5 | 관리자가 거절하면 **REJECTED**로 전환된다 |  |
| USER-001-6 | 관리자가 기존 활성 사용자를 비활성화하면 **DISABLED**로 전환된다 |  |

#### BP-USER-002: 정원 제한

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| USER-002-1 | **ACTIVE 사용자 기준으로 최대 4명**까지 허용한다 |  |
| USER-002-2 | ACTIVE 사용자가 4명인 상태에서 신규 가입 신청이 들어오면 \*\*"가입 불가(정원 초과)"\*\*로 처리한다 | 기본 정책 |
| USER-002-3 | (선택) PENDING은 생성하되 "승인 불가(정원 초과)" 상태로 표시할 수 있다 | 정책 설정 가능 |
| USER-002-4 | UI에 현재 ACTIVE 사용자 수 / 최대 정원(4)을 항상 표시한다 | Admin 페이지 |

#### BP-USER-003: 상태 전이

| 현재 상태 | 가능한 전이 | 조건 |
| --- | --- | --- |
| (신규) | → PENDING | 카카오 로그인 시 계정 없음 |
| (신규) | → ADMIN + ACTIVE | 최초 가입 (ACTIVE 0명) |
| PENDING | → ACTIVE | 관리자 승인 + 정원 미초과 |
| PENDING | → REJECTED | 관리자 거절 |
| ACTIVE | → DISABLED | 관리자 비활성화 |
| REJECTED | → (재가입 불가) | 별도 정책 없음 |
| DISABLED | → ACTIVE | 관리자 재활성화 (정원 미초과 시) |

### 4.3 예시

**정상 케이스:**

```
상황: 시스템에 사용자가 0명인 상태에서 첫 번째 사용자가 카카오 로그인한다
결과: 해당 사용자는 자동으로 ADMIN + ACTIVE 상태가 되며, 즉시 모든 기능을 사용할 수 있다
```

```
상황: ACTIVE 사용자가 2명인 상태에서 새 사용자가 가입하고 관리자가 승인한다
결과: PENDING → ACTIVE 전환. 현재 ACTIVE 3/4 표시
```

**경계 케이스:**

```
상황: ACTIVE 사용자가 4명인 상태에서 새 사용자가 카카오 로그인한다
결과: "정원 초과로 가입할 수 없습니다. 관리자에게 문의하세요." 메시지 표시. 가입 신청이 생성되지 않는다 (기본 정책)
```

```
상황: ACTIVE 사용자가 4명인 상태에서 관리자가 1명을 DISABLED로 전환한 후 PENDING 사용자를 승인한다
결과: ACTIVE 3명 → 승인 → ACTIVE 4명. 정상 처리
```

## 5. 추천 엔진 정책

### 5.1 목적

사용자의 선호시간대와 노선 설정에 따라 최적의 열차 후보를 검색·정렬하여 추천한다. SRT 직행을 최우선으로 하고, 필요 시 환승 대안을 제공한다.

### 5.2 규칙

#### BP-REC-001: 노선/역 설정

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| REC-001-1 | 기본 노선(Primary)은 **김천구미 ↔ 동탄**으로 고정한다 |  |
| REC-001-2 | 역 리스트는 내부 코드(id) + 표시명(name)을 분리하여 관리한다 | 확장 가능 설계 |
| REC-001-3 | `alternatives.enabled=true`이면 대체역을 후보 생성에 포함한다 | 환승/대체 도착역 |
| REC-001-4 | 대체역은 "환승 후보" 또는 "대체 도착/출발 제안"에 사용된다 |  |

#### BP-REC-002: 선호시간대 규칙

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| REC-002-1 | 방향(up/down) + 요일별로 **earliest_after** (N시 이후) 설정이 가능하다 |  |
| REC-002-2 | 해당 요일 00:00\~23:59 전체에서 **earliest_after 이후의 열차만** 후보로 허용한다 |  |
| REC-002-3 | 기본값 — 상행(김천→동탄): **금 18시 이후, 토 8시 이후** |  |
| REC-002-4 | 기본값 — 하행(동탄→김천): **일 12시 이후** |  |

#### BP-REC-003: 직행 후보 생성

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| REC-003-1 | **SRT 직행**을 최우선으로 후보를 생성한다 |  |
| REC-003-2 | 직행 후보 출력 수는 **Top N** (기본 5) | 설정 가능 |

#### BP-REC-004: 환승 후보 생성

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| REC-004-1 | 환승 후보는 **allow_transfer=true**일 때만 생성한다 |  |
| REC-004-2 | 최대 환승 횟수: **1회** |  |
| REC-004-3 | 최소 환승 여유시간: **20분** |  |
| REC-004-4 | 환승 후보 출력 수는 **Top N** (기본 3) | 설정 가능 |

#### BP-REC-005: 후보 수 제한

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| REC-005-1 | 방향별 후보 생성 최대치는 **30개**로 제한한다 | 조합 폭발 방지 |
| REC-005-2 | 대체역/환승 모두 활성화된 경우 반드시 상한을 적용한다 |  |

#### BP-REC-006: 스코어링/정렬

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| REC-006-1 | 기본 우선순위: ① SRT 직행(가산점) → ② 시간 조건 충족 → ③ 총 소요시간 짧음 → ④ 환승 횟수 적음 → ⑤ 환승 대기시간 짧음 |  |
| REC-006-2 | 가중치 설정 항목: `direct_bonus`, `total_time_penalty_per_min`, `transfer_penalty`, `transfer_wait_penalty_per_min` | 설정에서 조정 가능 |

### 5.3 예시

**정상 케이스:**

```
상황: 금요일 상행 검색, earliest_after=18시, allow_transfer=true
결과: 18:00 이후 SRT 직행 후보 최대 5개 + 환승 후보 최대 3개를 스코어링 정렬하여 출력
```

```
상황: 토요일 상행 검색, earliest_after=8시, allow_transfer=false
결과: 08:00 이후 SRT 직행 후보 최대 5개만 출력. 환승 후보는 생성하지 않음
```

**경계 케이스:**

```
상황: 일요일 하행 검색, earliest_after=12시, 12시 이후 SRT 직행이 0건
결과: 직행 후보 0건으로 표시. allow_transfer=true이면 환승 후보를 제공. 사용자에게 "직행 열차가 없습니다" 안내
```

```
상황: 대체역+환승 모두 활성, 후보 조합이 50개 발생
결과: 30개 상한 적용 후 스코어 상위 30개만 처리. 직행 Top 5 + 환승 Top 3 출력
```

## 6. 검색 범위 정책

### 6.1 목적

사용자가 1주일 단위가 아닌 복수 주(최대 8주)를 한 번에 검색할 수 있도록 하고, 이미 예매를 완료했거나 불필요한 주를 제외하여 효율적으로 추천을 받을 수 있게 한다.

### 6.2 규칙

#### BP-RANGE-001: 검색 기간 설정

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| RANGE-001-1 | 사용자가 **주 단위로 검색 범위를 설정**할 수 있다 |  |
| RANGE-001-2 | 범위: **최소 1주 \~ 최대 8주** (약 2개월) |  |
| RANGE-001-3 | 기본값: **1주일치** (이번 주 기준) |  |
| RANGE-001-4 | 검색 기준점: **이번 주부터(this)** 또는 **다음 주부터(next)** 선택 가능 |  |
| RANGE-001-5 | 설정된 주 수만큼 **연속으로 검색**을 수행한다 |  |

#### BP-RANGE-002: 주 단위 제외 (Skip Weeks)

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| RANGE-002-1 | 설정된 범위 내에서 **특정 주를 개별 제외**할 수 있다 |  |
| RANGE-002-2 | 제외 사유: 이미 예매 완료한 주, 이동이 필요 없는 주 (휴강, 공휴일 등) |  |
| RANGE-002-3 | UI에서 주 단위 캘린더/리스트로 **체크/해제** 방식으로 제외를 설정한다 |  |
| RANGE-002-4 | 제외된 주는 **검색/추천/알림 대상에서 완전히 제외**된다 |  |
| RANGE-002-5 | 제외된 주는 Dashboard/Results에서 **"건너뜀"** 태그로 표시한다 |  |
| RANGE-002-6 | 텔레그램 알림에도 제외된 주 정보를 요약 표기한다 |  |

#### BP-RANGE-003: 복수 주 결과 출력

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| RANGE-003-1 | 복수 주 검색 결과는 **주 단위로 그룹핑**하여 출력한다 |  |
| RANGE-003-2 | 각 주별로 기존 상행/하행 탭 → 직행/환승 섹션 구조를 유지한다 |  |
| RANGE-003-3 | UI에 **주 단위 네비게이션** (이전 주 / 다음 주 전환)을 제공한다 |  |
| RANGE-003-4 | 텔레그램: 주 단위로 메시지를 **분리 발송** (1주 = 1메시지) 또는 **전체 요약 1메시지**로 구성 가능 (설정 가능) |  |

#### BP-RANGE-004: 데이터 저장

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| RANGE-004-1 | 검색 범위 설정은 `runs` 테이블에 **search_range_weeks** (정수)로 저장한다 |  |
| RANGE-004-2 | 시작 기준점은 `runs` 테이블에 **start_from** (this/next)으로 저장한다 |  |
| RANGE-004-3 | 제외된 주 목록은 `runs` 테이블에 **excluded_weeks** (JSON 배열)로 저장한다 |  |

#### BP-RANGE-005: 주간 캘린더 관리

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| RANGE-005-1 | **월간/주간 캘린더 뷰**를 제공하여 향후 최대 8주를 한눈에 조망할 수 있다 | Calendar 페이지 |
| RANGE-005-2 | 각 주에 대해 **티켓 상태**를 관리한다: 필요(NEEDED) / 예매완료(BOOKED) / 불필요(NOT_NEEDED) / 검색중(SEARCHING) / 추천완료(RECOMMENDED) |  |
| RANGE-005-3 | 사용자 수동 설정 상태: **필요 / 예매완료 / 불필요** |  |
| RANGE-005-4 | 시스템 자동 전환 상태: **검색중** (실행 시작 시) / **추천완료** (실행 완료 시) |  |
| RANGE-005-5 | 실행 실패 시 해당 주 상태를 **"필요"로 롤백**하고 오류를 표시한다 |  |

#### BP-RANGE-006: 캘린더 검색 연동

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| RANGE-006-1 | **"필요" 상태인 주만** 검색 대상에 포함된다 | Skip Weeks 연동 |
| RANGE-006-2 | "예매완료" / "불필요"로 설정된 주는 검색에서 **자동 제외**된다 |  |
| RANGE-006-3 | 캘린더에서 선택한 주에 대해 **\[선택한 주 검색\]** 버튼으로 즉시 실행할 수 있다 |  |
| RANGE-006-4 | **\[전체 검색\]** 버튼으로 "필요" 상태 전체를 한 번에 실행할 수 있다 |  |
| RANGE-006-5 | 스케줄 실행 시에도 캘린더의 **"필요" 상태 주만** 대상으로 실행한다 |  |

#### BP-RANGE-007: 주별 메모

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| RANGE-007-1 | 각 주에 **짧은 메모**를 남길 수 있다 (예: "3/7 금 KTX 예매완료", "공휴일 주간") |  |
| RANGE-007-2 | 메모는 캘린더 뷰에 **미리보기 표시**된다 |  |
| RANGE-007-3 | 메모는 `week_plans` 테이블의 **memo** 필드에 저장한다 |  |

### 6.3 예시

**정상 케이스:**

```
상황: 사용자가 "이번 주부터 4주일치" 검색 범위를 설정하고 실행한다
결과: 이번 주 + 다음 주 + 2주 후 + 3주 후, 총 4주분의 열차를 검색. 주별로 그룹핑된 결과 출력
```

```
상황: 사용자가 "이번 주부터 4주일치" 중 2주차를 제외(이미 예매 완료)하고 실행한다
결과: 1주차, 3주차, 4주차만 검색. 2주차는 "건너뜀" 태그로 표시. 텔레그램에도 "2주차 제외" 표기
```

**캘린더 관리 케이스:**

```
상황: 사용자가 Calendar 페이지에서 3주차를 "예매완료"로 표시하고 메모에 "SRT 18:00 예매"를 입력한다
결과: 3주차 상태가 BOOKED로 전환. 이후 검색에서 3주차는 자동 제외. 캘린더에 메모가 표시됨
```

```
상황: 스케줄이 실행되며 캘린더에 "필요" 상태인 1주차, 4주차만 대상으로 검색한다
결과: 1주차, 4주차 상태가 "검색중"으로 전환 → 검색 완료 시 "추천완료"로 전환 + 결과 링크 제공
```

**경계 케이스:**

```
상황: 8주 범위를 설정하고 7개 주를 모두 제외하여 1개 주만 남긴다
결과: 해당 1개 주만 검색·추천. 정상 동작
```

```
상황: 모든 주가 "예매완료" 또는 "불필요"인 상태에서 [전체 검색]을 클릭한다
결과: "검색 대상 주가 없습니다. 캘린더에서 '필요' 상태 주를 확인해주세요." 안내 표시. 실행하지 않음
```

## 7. 알림/텔레그램 정책

### 7.1 목적

추천 결과를 텔레그램으로 적시에 전달하되, 동일 결과의 중복 발송을 방지한다. 복수 주 검색 시 주 단위 분리 발송을 지원한다.

### 7.2 규칙

#### BP-NOTI-001: 발송 내용

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| NOTI-001-1 | 제목: **방향 + 날짜 + 노선** (예: "🔼 상행 03/07(금) 김천구미→동탄") |  |
| NOTI-001-2 | 본문: **직행 후보 → 환승 후보** 순서로 출력 |  |
| NOTI-001-3 | 하단: **적용 조건 요약** (컷오프 시간/환승 허용 여부/대체역 등) |  |

#### BP-NOTI-002: 중복 방지 (dedupe)

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| NOTI-002-1 | 동일 결과 해시 기반으로 **180분 이내** 중복 발송을 금지한다 | dedupe_window_minutes 설정 가능 |
| NOTI-002-2 | dedupe 해시는 `dedupe` 테이블에 저장하며, `expires_at` 이후 자동 만료 |  |
| NOTI-002-3 | UI에 dedupe 상태 및 마지막 전송 시각을 표시한다 |  |

#### BP-NOTI-003: 테스트/재전송

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| NOTI-003-1 | **테스트 메시지 전송** 기능을 제공한다 (텔레그램 연결 확인용) |  |
| NOTI-003-2 | UI에서 특정 실행 결과를 **수동 재전송**할 수 있다 | dedupe 적용 |

### 7.3 예시

```
상황: 스케줄 실행 완료 후 텔레그램 발송이 트리거된다
결과: 결과 해시를 생성하고 dedupe 테이블 확인. 180분 내 동일 해시가 없으면 발송. 있으면 발송 건너뜀
```

```
상황: 사용자가 UI에서 [텔레그램 재전송] 버튼을 클릭한다
결과: dedupe 확인. 180분 내 동일 해시가 있으면 "최근 전송된 동일 결과가 있습니다" 안내. 강제 재전송 옵션 제공
```

## 8. 실행/스케줄 정책

### 8.1 목적

수동 실행과 스케줄 실행을 안정적으로 관리하고, 중복 실행을 방지한다. 검색 범위(주 수)와 제외 주 설정을 실행에 반영한다.

### 8.2 규칙

#### BP-EXEC-001: 수동 실행

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| EXEC-001-1 | UI에서 **검색 범위(주 수) + 시작점 + 제외 주**를 확인 후 **\[지금 검색\]** 클릭 시 즉시 실행 | BP-RANGE 연동 |
| EXEC-001-2 | 실행 중에는 버튼을 비활성화하고 **로딩/진행 상태**를 표시한다 |  |
| EXEC-001-3 | 실행 결과는 `runs` 테이블에 저장한다 (plan_json, result_json, status, 시간 등) |  |

#### BP-EXEC-002: 스케줄 실행

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| EXEC-002-1 | 스케줄은 **등록/활성/비활성/삭제**가 가능하다 |  |
| EXEC-002-2 | 크론 프리셋을 제공한다 (예: 매주 화 09:00) |  |
| EXEC-002-3 | 스케줄별로 시작점(**this_week / next_week**)과 \*\*검색 범위(주 수)\*\*를 설정할 수 있다 | BP-RANGE 연동 |
| EXEC-002-4 | 고급 크론 표현식 직접 입력을 지원한다 |  |

#### BP-EXEC-003: 중복 방지

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| EXEC-003-1 | 동일 시간대에 실행 요청이 중복되면 **하나만 수행**한다 (락 기반) |  |
| EXEC-003-2 | 수동 실행과 스케줄 실행이 동시에 트리거되면 먼저 시작된 것만 수행한다 |  |

### 8.3 예시

```
상황: 매주 화 09:00 스케줄이 설정되어 있고, 화요일 09:00에 사용자가 수동 실행도 클릭한다
결과: 먼저 시작된 실행만 수행. 두 번째 요청은 "이미 실행 중입니다" 메시지 표시
```

```
상황: 스케줄 실행 중 외부 API 오류로 실패한다
결과: runs.status='FAILED', 오류 상세 기록. 재시도 정책에 따라 대응 (BP-ERROR 참조)
```

### 9.3 예시


**경계 케이스:**

```

결과: 추가 자동예매를 중단. 텔레그램으로 "주간 결제 한도에 도달했습니다" 알림 발송. UI에도 표시
```

**에러 케이스:**

```
상황: auto 모드 실행 중 SRT 로그인이 3회 연속 실패한다
결과: 즉시 실행 중단. 텔레그램으로 "⚠️ 자동예매 중단: 로그인 반복 실패" 알림. auto.enabled는 유지하되 해당 실행은 FAILED 처리
```

**결제 수단 관리 케이스:**

```
상황: 관리자가 Safety 페이지에서 SRT 회원번호/비밀번호를 입력하고 저장한다
결과: 백엔드가 /data/.env.credentials 파일에 SRT_MEMBER_ID=xxx, SRT_MEMBER_PW=xxx 형태로 기록. UI에는 "SRT 계정: ●●●●1234 (등록됨)" 표시
```

```
상황: GET /api/credentials 요청 시
결과: { "srt": { "registered": true, "member_id_masked": "●●●●1234" }, "korail": { "registered": false } } 형태로 반환. 평문 비밀번호는 절대 포함하지 않음
```

```
상황: auto 모드 실행 시 SRT 계정이 미등록 상태이다
결과: "SRT 계정이 등록되지 않았습니다. Safety 페이지에서 계정을 먼저 등록해주세요." 안내. 실행하지 않음
```

## 10. 오류 처리 정책

### 10.1 목적

외부 시스템(SRT/KTX 조회 API) 장애 시에도 서비스 안정성을 유지하고, 사용자에게 적절한 피드백을 제공한다.

### 10.2 규칙

#### BP-ERROR-001: 외부 조회 실패

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| ERROR-001-1 | 외부 조회 실패 시 **graceful error** 처리: 로그 기록 + UI 표시 + 필요 시 텔레그램 알림 |  |
| ERROR-001-2 | 레이트리밋/재시도 정책을 적용한다 (지수 백오프 권장) |  |

#### BP-ERROR-002: 실행 실패

| 규칙 ID | 규칙 | 비고 |
| --- | --- | --- |
| ERROR-002-1 | 실행 실패 시 runs.status='FAILED', 오류 상세를 result_json에 기록한다 |  |
| ERROR-002-2 | Logs 페이지에서 오류 상세를 확인할 수 있다 |  |

### 10.3 예시

```
상황: 스케줄 실행 시 SRT 조회 API가 503 응답을 반환한다

```
